**CVOYA Graph 1.0 is the first stable release of CVOYA's type-safe graph library for .NET 10.**

CVOYA Graph is an open-source library for applications that work with connected data. In a graph,
the things in a domain are represented as nodes and the connections between them as relationships:
people and who they know, services and their dependencies, products and their supply chains, or any
other model where the connections matter as much as the records themselves.

The library lets you describe that graph with ordinary C# types, query and traverse it through a
LINQ-based API, and choose a storage provider without rewriting the application around a different
public programming model.

## Work with graphs in ordinary C#

Nodes and relationships are records or classes in your own domain model. CVOYA Graph does not
require every entity to inherit a database ID, and relationship objects do not have to carry
endpoint IDs or direction fields. Add a domain key when your application has one, use multiple
properties for a composite key, or leave an entity keyless.

From that model, an application can:

- Create nodes and relationships together as one atomic operation.
- Query strongly typed nodes and relationships with LINQ, or use dynamic queries when the shape is
  only known at runtime.
- Follow relationships, inspect path segments, and return complete paths through the graph.
- Search text, group and aggregate results, and stream large result sets asynchronously.
- Update or delete an entire query selection with one set-based operation.
- Group related work in transactions.

Optional compile-time analyzers catch invalid graph models while you build, and an incremental
source generator creates the serialization code used to move domain objects to and from a
provider.

## Choose where the graph runs

CVOYA Graph 1.0 includes three providers behind the same core API:

- **Neo4j** is the complete database-backed reference provider for production applications.
- **PostgreSQL with Apache AGE 1.7** stores the graph inside PostgreSQL and supports most of the
  shared 1.0 contract.
- **In-memory** runs entirely inside the application. It is a deterministic test double and a
  working reference for provider behavior, not a production database.

The database providers work with native graph storage. They can query compatible nodes and
relationships created outside CVOYA Graph without requiring CVOYA-specific IDs, root labels,
discriminators, or relationship metadata.

Provider authors can implement the same public contracts and use the packaged compatibility suite
to verify how their provider behaves.

## Get started

Install the provider for your database from the
[CVOYA NuGet profile](https://www.nuget.org/profiles/cvoya); the provider brings in the core and
serialization packages it needs. The optional `Cvoya.Graph.Analyzers` package adds compile-time
model checks.

Follow the [quick start](https://github.com/cvoya-com/graph/blob/v1.0.0/README.md#quick-start) to
define a small graph, connect it to Neo4j, create data, and run a traversal. The
[complete documentation](https://oss.cvoya.com/graph/) covers modeling, querying, transactions,
testing, provider behavior, and extending the library.

## Provider-specific differences

Neo4j implements the complete 1.0 query and transaction surface. Apache AGE does not currently
support nested transactions, ordering by a whole entity, or shortest path queries. CVOYA Graph
reports those operations as unsupported instead of silently substituting behavior with different
semantics. The in-memory provider does not support full-text indexes or whole-entity ordering; its
text search is an index-free whole-token scan intended for tests.

## If you evaluated a prerelease build

If 1.0 is your first CVOYA Graph installation, no migration is required. If you used an alpha or
preview build, the stable `Cvoya.Graph.*` packages replace the earlier `Cvoya.Graph.Model.*`
package line, and the final storage model is intentionally different.

CVOYA Graph 1.0 does not read or rewrite alpha-era universal IDs, relationship endpoint fields,
root labels, or provider metadata. Back up prerelease data, update the application model, then
**recreate and reimport** the database or perform your own reviewed transformation. See the
[v1 migration guide](https://github.com/cvoya-com/graph/blob/v1.0.0/docs/migration-0.x.md).
